B-124 — Dashboard web enriquecido com despachos e localizacoes

Ω-GOV (2026-07-13): politica KPI-por-PR (D-KPI-PER-PR) substitui a publicacao pos-avaliacao humana. Backend 766/766 (suite backend INTEIRA no gate do CI apos o Ω-GATE; antes 15/15 media so core-saas). Demais metricas nos ultimos valores oficiais (B-124, web/backend/docs-only): Flutter 764/764, smoke web 44/44 (era 33/33), contratos mobile 18/18, contratos mobile + Core SaaS 21/21, modulos Flutter 17/17, MVP demo 96%, MVP vendavel 78% e 49 blocos entregues.

B-124 (web-only): o Dashboard web passou a compor 4 fontes reais em paralelo — GET /work-orders + GET /operations/dispatches + GET /field-locations/latest + GET /notifications/unread-count (+ GET /approvals/pending) — com 8 KPIs derivados, fila critica combinada (SLA vencido > prioridade > operador sem sinal > aprovacao pendente > OS sem operador), despachos ativos, status de campo real (regra stale de 15 min reutilizada do operations-map.adapter), alertas acionaveis e eventos derivados das listas carregadas (sem timeline por OS). Fallback por fonte com rotulos Dados demonstrativos/Fallback local; nenhum arquivo mobile/backend alterado.

B-123 (visual-only): 7 telas/areas do fluxo de OS mobile alinhadas ao prototipo aprovado — lista de OS, detalhe/check-in, execucao, checklists da OS, execucao de checklist, evidencias e sincronizacao/fila offline — com estados semanticos por tokens centrais e sem dado tecnico cru; nenhum repository/service/ contrato/sync/model/provider alterado.

B-122: Perfil do operador recriado fiel ao prototipo aprovado (screen-refs/mobile/perfil.png) — sem token, modo de autenticacao, permissoes cruas ou IDs na UI; auditoria das telas MVP registrada.

Web MVP ligado aos endpoints reais: lista/detalhe de OS com timeline, Dashboard composto de work-orders + notifications, Aprovacao operacional no detalhe (motivo obrigatorio na reprovacao, RBAC) e navegacao MVP-only via GET /navigation/menu.

Hardening mobile: timeline real no detalhe/check-in com fallback local seguro, auto-sync montado no app root com ordem segura, adapter de checklist tolerando fields e components (tipo desconhecido mostra mensagem segura) e base URL por --dart-define=API_BASE_URL.

Limitacoes: S3/presigned real, DB/Redis receipt, antivirus real, download protegido final, retencao definitiva e Settings web sem backend dedicado. (Dashboard web sem dispatches/field-locations foi resolvido no B-124.)

Politica de KPIs duplos: mexeu Flutter/mobile atualiza mobile/flutter_app/Kpis/* e reflete percentuais em Kpis/*; mexeu fora mobile atualiza Kpis/*; mexeu nos dois atualiza ambos; se existir index.html, o HTML tambem deve ser atualizado.

Política permanente de KPIs pós-avaliação humana: PRs de feature não devem atualizar arquivos de KPI; devem reportar KPIs propostos apenas no relatório final. KPIs só devem ser publicados após avaliação humana, merge e gate, em bloco documental/KPI separado.
