Handoff — MVP Mobile

ERP Techsolutions · App de Campo (Flutter) · 11 telas MVP · v2026.06

Escopo: somente o MVP Mobile (11 telas de operação de campo). Este documento concilia a especificação do orquestrador com o que já está construído no protótipo ERP Mobile.dc.html — porque o orquestrador não está ciente de todas as mudanças de lógica que fizemos. Cada tela traz a spec, os endpoints e um bloco "No protótipo" destacando o que mudou, foi estendido ou ainda falta.

Marcadores de reconciliação: CONFORME como na spec ALTERADO construído diferente / estendido EXTRA existe no protótipo, fora da spec PENDENTE na spec, ainda não/parcial
Contexto de integração — ler primeiro

Todos os dados do protótipo são mock estruturado na classe de lógica (class Component). Nenhum endpoint mobile está conectado à rede ainda — as rotas abaixo são o contrato a implementar no Flutter. Ações de escrita (check-in, checklist, assinatura, conclusão) hoje só mudam o estado local. As três coisas que mais divergem da spec do orquestrador: (1) guincho tem coleta e entrega como execuções separadas; (2) o check-in valida por 2 dígitos (placa no guincho / nº de série no prestador) + geolocalização; (3) o prestador tem sub-fluxo Diagnóstico → Execução → Estoque do técnico.

Convenções (Mobile)

Padrão de plataforma
SessãoPOST /auth/login → JWT access+refresh · POST /auth/refresh · POST /auth/logout · header Authorization: Bearer + X-Tenant-Id.
BootstrapGET /mobile/bootstrap devolve usuário, organizações, papel, módulos e flags — origem da Seleção de Organização e da Home.
Offline-firstEscrita em fila local (Hive/Isar/SQLite) → replay via /mobile/sync/*. Conflito = 409 com as duas versões; resolução manual, nada sobrescrito.
LocalizaçãoSó manual — POST /mobile/field-locations. Sem background tracking, timer ou stream. Consentimento LGPD obrigatório (retenção 90d, revogável).
MascaramentoPlaca e nº de série exibidos com os 2 últimos dígitos como **; o operador digita os 2 dígitos na chegada para liberar "Iniciar atendimento" (validação local + geo).
ChecklistSchema-driven — o modelo vem do backend por cliente (/checklists/:id/render); o app renderiza os componentes dinamicamente. Run persistido com markers/attachments/complete.

No protótipo — o que mudamos
  • Coleta e entrega são runs separados: o estado usa prefixos c_ (coleta) e e_ (entrega) — cada um com seu damage_map (mapa de avarias) e sua assinatura.
  • A assinatura da entrega é distinta da coleta: título "Assinatura de quem recebeu" e canvas próprio; ambas persistem ao reabrir (localStorage no protótipo → storage seguro no Flutter).
  • photo_upload = grid de 8 miniaturas: só aparecem com foto capturada; a próxima vaga mostra o ícone de câmera; tocar numa foto já feita abre "capturar nova foto".
  • Botão "Finalizar" habilita só com pendências = 0 (derivado dos itens obrigatórios); a barra de progresso atualiza por passo.

Pendências para o MVP
  • Tela 7 (lista de checklists) — não construída; decidir se entra como hub ou fica como "abrir direto" no MVP.
  • OAuth Google e biometria — hoje mockados/como preferência; precisam de fluxo real.
  • Todos os endpoints estão como contrato — nenhuma chamada de rede real no protótipo; ações escrevem só no estado local.
  • Definir schema real do checklist por cliente (o backend modela; o app já renderiza os 7 componentes).
